简版GIT入门教程

发表于 2017-06-13 | 分类于 Programming ， GitHub |

因为最近在学习R和统计，以及前一阵在GitHub上搭建博客，不可避免接触GIT版本管理。相对于R、统计和HEXO来说，GIT的学习一直在计划中，但又一直懒得去碰，直到最近碰到了一个GIT的具体问题，不得不硬着头皮上。幸好有廖雪峰的博客及其GIT教程，堪称GIT入门经典！不由得佩服廖老师对GIT的理解以及写作功底，在前面栽了好大一棵树。借用GIT教程开篇的一句话，“牛是怎么定义的呢？大家可以体会一下。”

将教程浓缩成提纲式备忘录，便于回顾。以下仅适用于回顾，初次学习应该直接学习廖雪峰的GIT教程。

配置GIT

首次安装配置

以Windows版本Git为例，按默认选项安装。安装完成后，在Git Bash命令行输入进行配置：

  • 因为Git是分布式版本控制系统，所以，每个机器都必须自报家门：你的名字和Email地址。你也许会担心，如果有人故意冒充别人怎么办？这个不必担心，首先我们相信大家都是善良无知的群众，其次，真的有冒充的也是有办法可查的。

  • 注意git config命令的--global参数，用了这个参数，表示你这台机器上所有的Git仓库都会使用这个配置，当然也可以对某个仓库指定不同的用户名和Email地址。

创建空目录

如果使用Windows系统，请确保目录名（包括父目录）不包含中文。

创建版本库

  • 通过git init命令把这个目录变成Git可以管理的仓库。仓库建好后，GIT告诉你是一个空的仓库
  • 当前目录下多了一个.git的目录，这个目录是Git来跟踪管理版本库的，没事千万不要手动修改这个目录里面的文件，否则就把Git仓库给破坏了。
  • 也不一定必须在空目录下创建Git仓库，选择一个已经有东西的目录也是可以的。

版本修改操作

工作区与版本库

  • 工作区（Working Directory）：就是在电脑里能看到的目录，比如learngit文件夹
  • 版本库（Repository）：工作区有一个隐藏目录.git，它是GIT的版本库。版本库中包含暂存区（stage）和分支（branch）。

版本更新

  • 用命令git add告诉Git，把文件添加到仓库。
  • 用命令git commit告诉Git，把文件提交到仓库。每次修改，如果不add到暂存区，那就不会加入到commit中。
  • -m后面输入的是本次提交的说明
  • rm命令用于删除文件

修改状态查询

git status

git status命令可以让我们了解仓库当前的状态，修改尚未被add状态Changes not staged for commit，或者已经被add等待被commit状态Changes to be committed，或者已经commit完成nothing to commit (working directory clean)

git diff <file>

如果git status告诉你有文件被修改过，git diff可以查看修改内容。

git log

  • 命令git log显示从最近到最远的提交日志，3628164...882e1e0的是commit id（版本号）
  • 命令git reflog用来记录你的每一次命令，以防版本回退后无法从git log中获取版本号

版本回退

每当你觉得文件修改到一定程度的时候，就可以“保存一个快照”，这个快照在Git中被称为commit。每次commit提交，Git都把它们串成一条时间线（这条时间线就是一个分支）。一旦你把文件改乱了，或者误删了文件，还可以从最近的一个commit恢复，然后继续工作，而不是把几个月的工作成果全部丢失。GIT中，HEAD指向的版本就是当前版本（），因此，GIT允许我们在版本的历史之间穿梭，使用命令git reset --hard commit_id。

  • 在Git中，用HEAD表示当前版本，上一个版本就是HEAD^，上上一个版本就是HEAD^^，前第100个版本是HEAD~100
  • --hard
  • 也可以恢复到指定版本，向前向后皆可。使用版本号恢复时，版本号不必要写全。
  • 回退版本之后再使用git log，HEAD指向版本之后的版本将不出现在log中。

撤销修改

撤销工作区修改

git checkout -- file可以丢弃工作区的修改，使文件回到最近一次git commit或git add时的状态。

  • git checkout其实是用版本库里的版本替换工作区的版本
  • 如果readme.txt自修改后还没有被放到暂存区，撤销修改就回到和版本库一模一样的状态；
  • 如果readme.txt已经添加到暂存区后，又作了修改，撤销修改就回到添加到暂存区后的状态。
  • git checkout -- file命令中的--很重要，没有--，就变成了“切换到另一个分支”的命令。

撤销暂存区修改

git reset HEAD file可以把暂存区的修改撤销掉（unstage），重新放回工作区，而不会直接删除修改。如果要进一步删除工作区修改，应使用git checkout -- file命令。git reset命令既可以回退版本，也可以把暂存区的修改回退到工作区。当我们用HEAD时，表示最新的版本。

撤销本次提交

此时只能通过版本回退，不过前提是没有推送到远程库。

远程仓库

配置SSH

本地Git仓库和GitHub仓库之间的传输是通过SSH加密的。为什么GitHub需要SSH Key呢？因为GitHub需要识别出你推送的提交确实是你推送的，而不是别人冒充的，所以GitHub只要知道了你的公钥，就可以确认只有你自己才能推送。当然，GitHub允许你添加多个Key。假定你有若干电脑，你一会儿在公司提交，一会儿在家里提交，只要把每台电脑的Key都添加到GitHub，就可以在每台电脑上往GitHub推送了。

配置SSH详细操作可参考这一篇文章。

关联本地与远程分支

  • 用git push命令把本地库的内容推送到远程，实际上是把当前分支master推送到远程。
  • 第一次推送时，由于远程库是空的，我们加上了-u参数，这样可以把本地的master分支和远程的master分支关联起来。

正常推送

  • 每次本地提交后，只要有必要，就可以使用命令git push origin master推送最新修改；
  • 关联分支之后，在以后的推送或者拉取时就可以简化命令，不需要-u参数。

参与GitHub开源项目

  • 访问项目主页，“Fork”该项目到自己的账号，克隆该项目仓库到GitHub
  • 从自己的GitHub克隆到本地，git clone git@github.com:michaelliao/bootstrap.git
  • 本地干完活后，往自己的GitHub仓库推送。
  • 如果你希望开源项目官方库能接受你的修改，你就可以在GitHub上发起一个pull request。

分支管理

分支简介

在Git里，每次提交Git都把它们串成一条时间线，这条时间线就是一个分支。

  • 当仓库只有一个分支时，它称为主分支或master分支。可以根据需要创建其他相关分支，比如dev分支，Bug分支，Feature分支。
  • master和dev既是该分支名，也是该分支的指针。如，dev指针指向该分支的某一版本号（某一次提交），HEAD再指向dev，GIT就能确定当前分支在dev上，以及当前分支的提交点。
  • 一般来说开发在dev分支，dev完成后，就可以把dev合并到master上。

在实际开发中，我们应该按照几个基本原则进行分支管理：

  • 首先，master分支应该是非常稳定的，也就是仅用来发布新版本，平时不能在上面干活；
  • 那在哪干活呢？干活都在dev分支上，dev分支是不稳定的，到某个时候，再把dev分支合并到master上，在master分支发布新版本；
  • 每个人都在dev分支上干活，每个人都有自己的分支，时不时地往dev分支上合并就可以了。

创建与合并分支

分支操作命令汇总：

  • 查看分支：git branch
  • 创建分支：git branch <name>
  • 切换分支：git checkout <name>
  • 创建+切换分支：git checkout -b <name>
  • 合并某分支到当前分支：git merge <name>
  • 删除分支：git branch -d <name>
  • 删除一个没有被合并过的分支：git branch -D <name>强行删除

分支合并策略

分支合并最简单的方法，就是直接把master指向dev的当前提交，即Fast-forward。但这种模式下，删除分支后，会丢掉分支信息。

如果要强制禁用Fast forward模式，Git就会在merge时生成一个新的commit，这样，从分支历史上就可以看出分支信息。准备合并dev分支，请注意--no-ff参数，表示禁用Fast forward。因为本次合并要创建一个新的commit，所以加上-m参数，把commit描述写进去。合并后，我们用$ git log --graph --pretty=oneline --abbrev-commit查看分支历史。

Bug分支

  • 修复bug时，我们会通过创建新的bug分支进行修复，然后合并，最后删除。
  • 当手头工作没有完成时，先把工作现场git stash一下，然后去修复bug，修复后，再git stash pop，回到工作现场。

如果恢复stash内容后，并不想删除stash内容，你需要用git stash apply恢复，而不是git stash pop。

  • 当想删除数据时，用git stash drop来删除。
  • 可以多次stash，这样就会保存多份stash内容。恢复的时候，先用git stash list查看，然后恢复指定的stash，$ git stash apply stash@{0}

自定义GIT

忽略特殊文件

在Git工作区的根目录下创建一个特殊的.gitignore文件，然后把要忽略的文件名填进去，GIT在提交更改时就会自动忽略这些文件。需要注意的是，.gitignore文件本身要放到版本库里，它所覆盖的是整个版本库。.gitignore文件不需要从头写，GitHub已经为我们准备了各种配置文件，只需要组合一下就可以使用了。所有配置文件可以直接在线浏览：https://github.com/github/gitignore

最后一步就是把.gitignore也提交到Git，$ git add .gitignore，就完成了！当然检验.gitignore的标准是git status命令是不是说working directory clean。
